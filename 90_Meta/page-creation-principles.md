# ページ作成原則 — LLM Wiki パターンの実務適用

welfare-graph における **ページ（ノード）作成** の判断基準。Karpathy 方式「LLM Wiki」パターンを **障害福祉領域の相談支援実務** に適用するための運用原則。

## 🪷 Mantra（合言葉）

> **「いつか必要」は「今は不要」と同義**

ページを作りたくなった瞬間に、この一言を反芻する。投機的作成の 8 割はこの mantra で防げる。

---

## §1 なぜ投機的作成を禁じるのか

### 1.1 LLM Wiki パターンの考え方

従来型の知識管理（Zettelkasten・PKM 等）は「網羅的な知識ベースを構築することで後から検索しやすくする」という発想。一方 LLM Wiki パターンは次の設計思想:

> **実際に使った知識だけを構造化して堆積させる**

需要駆動型の設計思想。つまり:

- グラフは **構築する成果物** ではなく **実務の残渣・堆積物**
- 庭（実務）の手入れで出る **堆肥** であって、庭仕事そのものではない
- 法律書を全部書き写した Wiki より、**利用者 P-XXXX のアセスメントで実際に参照した法令の抜粋** のほうが価値が高い

### 1.2 投機的作成の害

「グラフが断線している」「参照先が欠落している」という理由で先回り作成すると:

| 害 | 具体例 |
|---|---|
| **検索ノイズ** | MCP サーバの `search_vault("高次脳機能障害")` で、実利用者ページ P-XXXX と無関係な「教科書的定義」がヒット |
| **陳腐化の不可視化** | 使われていないため、法改正等で古い情報のまま放置される（特に報酬改定・雇用率・法定期日が変動しやすい法令） |
| **メンテコスト増** | `review_due` 期限管理が増え、未使用のまま腐る確率が高い |
| **粒度のズレ** | 実務で本当に必要な論点と、教科書的網羅性の間に乖離が生じる |
| **出典不在** | 記憶・一般知識で書かれたページは出典検証ができず、**誤情報混入リスク** |

### 1.3 壊れリンクの設計上の役割

Obsidian の **未解決リンク**（薄灰色表示）は LLM Wiki において:

1. **需要の可視化**: グラフビューで複数エッジを持つ薄灰色ノードは「**昇格待ち** の有力候補」として可視化される
2. **強制ではない**: 参照したい時に wikilink は書ける、でも中身を書く義務はない
3. **記述の節度**: 書き手が「今書かなくてもいい」と思える精神的余裕を与える

→ **壊れリンクは失敗ではなく、設計されたバッファ**。

## §2 ページ作成前の 3 つの関門

ページを作りたくなった瞬間、次の 3 つを自問する。**1 つでも No なら作らない**。

### Q1. 今日の実務で、このページがあれば助かった場面はあったか？

- ✅ **Yes** → 作成候補。どの業務場面だったかを本文冒頭の `triggered_by` に記録
- ❌ **No** → 作らない
- ⚠️ 「**将来あるかもしれない**」→ No と同義

### Q2. 原典（raw/）が手元にあるか？

- ✅ **Yes** → ingest として作成（条文番号・出典 URL を frontmatter に保存）
- ❌ **No（記憶・一般知識で書く）** → 作らない
  - LLM Wiki パターンの要諦は **出典に紐づく知識の蓄積** であり、教科書的定義の再生産ではない
  - 原典がないのにページを作りたいなら、まず raw/ に資料を配置することから始める

### Q3. 作らなかったら、次に誰が・いつ困るか？

具体的に答えられなければ作らない:

- ✅ 「来週 P-0002 の就学相談で参照する」「5/15 の研修資料に使う」
- ❌ 「将来の利用者のため」「グラフが完結するため」（需要が特定できていない）

## §3 作成時の必須手続き

作成が正当化される場合でも、以下を守る:

### 3.1 frontmatter の必須情報

```yaml
---
type: {law/guideline/framework/disorder/method/...}
source: "raw/..."                           # 必須: 原典パス
source_url: "https://..."                   # 必須: 一次情報 URL
triggered_by: "[[20_Episodes/YYYY-MM-DD_P-XXXX]]"  # 推奨: 作成トリガーの具体エピソード
created: YYYY-MM-DD
updated: YYYY-MM-DD
status: active | provisional
---
```

`triggered_by` フィールド（新設提案）は **投機的作成との区別子** として機能する。書けなければ投機的作成の疑いが濃い。

### 3.2 本文冒頭の作成コンテキスト

```markdown
> [!note] 作成コンテキスト
> 2026-05-05、P-0002（30代精神障害・就労継続A型利用）の事業所から
> 「照明過敏の訴えが継続的に出ているが対応に困っている」との相談を受けたのが発端。
> 感覚過敏に関する配慮提供の判断枠組みとして整理。
> 類似事例（P-0005 等）にも展開可能。
```

→ **後日このページを見返した時に、なぜ作ったか・何に使えるかが即座にわかる**。

### 3.3 粒度の節度

- 初回は **論点特化** で書く（100-150 行目安）
- 教科書的網羅性を最初から目指さない
- 歴史的背景・国際法史・網羅的判例一覧は **別の実需要** が発生した時点で追記
- 「各セクションの裏に具体的な実務エピソード」を保つ

## §4 昇格プロセス（壊れリンク → ページ）

### 4.1 典型的な昇格パス

```
Day 0:  raw に資料が無く、実需要も無い → 壊れリンクのまま
Day N:  実務で同じ論点が 2-3 回参照される → 需要の実体化
Day N:  raw/ に原典を配置              → 原典の整備
Day N:  ingest 指示                    → ページ生成
Day N+Δ: 別の実需要が発生              → セクション追記（成長）
```

### 4.2 昇格判定のヒューリスティック

| 状況 | 判断 |
|---|---|
| 参照 1 件・raw なし | 壊れリンク維持 |
| 参照 1 件・raw あり・実需要あり | 昇格可 |
| 参照 2 件以上・同じ論点が再出現 | **強く昇格推奨**（DRY 違反の解消） |
| 参照 5 件以上・raw なし | raw/ 整備を先行させる |
| 参照 0 件・raw のみあり | raw/ に眠らせておく（ingest 不要） |

### 4.3 「原典の入手」を ingest の事前条件に

**原典がない場合は ingest しない**。実需要があるなら:

1. e-Gov / 厚労省 / 内閣府等から PDF をダウンロード
2. 該当する学術論文を探す
3. 実務ガイドライン・マニュアルを取り寄せる
4. それらを `raw/{laws,guidelines,reports,insights}/` に配置
5. **その後** ingest 指示

これを怠ると、記憶・LLM の訓練データベースからの再生産になり、**出典検証が不可能** なページが生まれる。

## §5 間引き・降格・archived

### 5.1 週次レビュー（10-15 分）

1. `raw/` を覗く → 1 週間で貯まった資料を見る
2. **実際に参照したものだけ** ingest 候補にする
3. 残りは raw/ に眠らせたまま（ingest 強迫を捨てる）

### 5.2 月次レビュー — 未使用ページの間引き

`vault_health_check.py` で検知したい指標（将来拡張）:

- 過去 30 日間で git log に参照・編集がないノート
- relations の **被参照が 0** のまま作成から 60 日経過したノート
- `triggered_by` が空のノート → 投機的作成の疑い

### 5.3 降格・archived

いきなり削除せず段階的に:

```
active → provisional（30日静観）→ archived/ に退避
```

- `status: provisional` は「需要駆動ではない暫定ページ」を示すラベル
- 実需要が発生すれば `active` に戻る
- `archived/` からも git 履歴で復元可能

## §6 例外 — 投機的作成が許容されるケース

以下は例外として投機的作成が許容される:

| ケース | 理由 |
|---|---|
| **MOC**（00_MOC/） | ナビゲーション構造。実体的知識ではなく配線図 |
| **90_Meta/ 設計書** | 運用設計・スキーマ・スクリプト。メタ層は需要駆動の対象外 |
| **層の README.md** | 運用方針を書く場所で、知識エンティティではない |
| **SCHEMA 辞書に定義された type の template** | `90_Meta/templates/` は鋳型 |

## §7 セルフチェックリスト（ページ作成前）

ノード作成時にこのリストを必ず通す:

- [ ] `raw/` に原典が配置されているか？
- [ ] `triggered_by` に書ける具体的なエピソードがあるか？
- [ ] 「3 か月後にこのページが存在しなかったら、誰がどう困るか」即答できるか？
- [ ] 今日の実務で実際に参照したか、または今週参照する予定か？
- [ ] 「グラフが完結する」「参照が断線しない」以外の理由があるか？
- [ ] 初回の粒度は論点特化（100-150 行目安）に収まっているか？
- [ ] 教科書的網羅性に引きずられていないか？

**6 つ以上 Yes** → 作成してよい
**5 つ以下** → 壊れリンクのまま残す or raw/ 整備から始める
